## 2026-05-19

### Fixed

- **`Badge` soft variants are visibly more present on dark page / sidebar backgrounds.** [packages/registry/registry/default/badge/badge.tsx](packages/registry/registry/default/badge/badge.tsx) — the dark-mode chip bg for the `success`, `warning`, and `info` variants was bumped from `bg-X/20` to `bg-X/30`. A 20% chroma tint over `--background` (`stone-800`, `#1c2229`) and especially over `--sidebar-background` (`stone-900`, `#11161b`) produced ~1.3–1.5:1 chip-vs-page contrast — chips with chroma close to the bg luminance (most notably `success` green) faded into the surface. Bumping to 30% raises the chip silhouette to ~1.8–2.0:1 in both contexts while staying clearly "soft" (not a solid badge). Text-on-chip contrast was already comfortably above 4.5:1 with the soft-fg tokens — the more saturated bg only widens that margin. The `muted` variant also gained a dark-mode bg bump: `dark:bg-stone-600/60 dark:text-stone-200` replaces the previous `dark:text-stone-300` (the original `bg-muted` resolves to `stone-700` in dark, which sat ~1.3:1 against the page bg and effectively disappeared on the sidebar). Light-mode behavior is unchanged for every variant. Non-breaking.

## 2026-05-18

### Added

- **`AppShell` branding gains an optional `subtitle` slot.** [packages/registry/registry/default/app-shell/app-shell-types.ts](packages/registry/registry/default/app-shell/app-shell-types.ts), [packages/registry/registry/default/app-shell/app-shell.tsx](packages/registry/registry/default/app-shell/app-shell.tsx) — `AppShellProps['branding']` adds `subtitle?: string`. When provided, the `BrandingLink` renders a second line under `branding.name` using `text-xs text-muted-foreground`, mirroring the existing `user.email` pattern in the user menu below the nav. The line lives inside the same `grid flex-1 ... leading-tight` block as the brand name, so the inherited `SidebarMenuButton` icon-collapse classes (`group-data-[collapsible=icon]:size-8! group-data-[collapsible=icon]:p-2!`) clip the text away when the sidebar shrinks to icon mode — no extra responsive logic needed in the component. Intended for short, non-interactive metadata such as a build/version string (`v0.0.7`), an environment tag (`staging`), or a tenant label. Long strings truncate via the existing `truncate` class. Contrast unchanged — `text-muted-foreground` over the sidebar background already clears AA in both themes. Non-breaking; `subtitle` is optional and unset call sites render exactly as before.

### Changed (BREAKING)

- **`CardTitle` default `size` is now `sm` (16px) instead of `lg` (24px).** [packages/registry/registry/default/card/card.tsx](packages/registry/registry/default/card/card.tsx) — the `size` prop is unchanged (`sm` | `md` | `lg`), but the default is flipped so that bare `<CardTitle>…</CardTitle>` calls render at the SaaS-default 16px product-card title weight instead of the 24px hero title. This matches the de-facto sizing used by the vast majority of product cards across consumers and reduces the per-card boilerplate (no more `size="sm"` on every product card). Hero / single-card surfaces that want the previous look must now opt in with `size="lg"`. No visual change for callers that already passed `size` explicitly. Contrast unchanged — typography only. The JSDoc on `CardTitleProps['size']` is updated so editor hover help reflects the new default.

### Changed

- **`SelectPicker` (multi mode) gains a `collapseSelectedAt` threshold.** [packages/registry/registry/default/select-picker/select-picker.tsx](packages/registry/registry/default/select-picker/select-picker.tsx) — `MultipleSelectPickerProps` adds `collapseSelectedAt?: number`. When the number of currently-selected options is strictly greater than the threshold, the trigger swaps the per-item removable `Badge` chips for a single `Badge variant="secondary"` reading `{N} selected`. The single badge uses the same classes (`rounded-sm px-1.5 py-0 text-xs font-normal`) as the existing chips so the trigger height does not change between the expanded and collapsed states. The badge intentionally has no × remove button — when collapsed, individual removal is done by opening the popover and toggling items off; this keeps the trigger from acting like a chip-management surface once the selection count grows. `selectedOptions.length === 0` still renders the placeholder, so the collapse logic only fires once at least one item is selected. Single-mode behaviour is untouched. Default behaviour is unchanged for callers that omit the prop (the value is `undefined`, so the collapse branch is skipped). Contrast unchanged — the single badge reuses the existing `secondary` variant, which already clears AA in both themes. Non-breaking.

- **`Avatar` root now paints a `bg-muted` base layer and `AvatarImage` renders a self-blurred backdrop at `md`+ sizes.** [packages/registry/registry/default/avatar/avatar.tsx](packages/registry/registry/default/avatar/avatar.tsx) — fixes the long-standing case where transparent-PNG avatars bled the page background through the avatar shape (the "floating character on the page" look). Three layered changes: (1) the `AvatarPrimitive.Root` class now includes `bg-muted` so any transparent region of an image sits on a clean muted circle instead of the page. (2) `AvatarImage` reads the parent Avatar size from a new `AvatarSizeContext`; when the size is `md`, `lg`, `xl`, or the legacy `h-10 w-10` default (no `size` set), it renders a second copy of the same image as an ambient backdrop layer: `absolute inset-0 scale-[1.75] object-cover blur-md saturate-50 opacity-60 pointer-events-none aria-hidden`. The browser cache deduplicates the second image so there's no extra network request. The blurred copy is desaturated and dimmed so the crisp foreground stays visually dominant — borrowed from the Apple-Music / Spotify art-as-background pattern. (3) `xs` and `sm` avatars skip the backdrop layer — the blur is invisible at 20–28px and the extra image decode is wasted work in dense lists. The colorized `AvatarFallback` is unchanged: when no image is loaded the existing per-character OKLCH hue still wins, so the no-image path looks exactly as before. The new `relative z-10` on the foreground image and `relative z-20` on the Fallback keep the stacking order explicit (Root bg < backdrop < foreground image < Fallback when shown). Non-breaking.

- **`AvatarGroup` adds a 2px hairline border on top of the existing background ring so stacked avatars read as separated circles.** [packages/registry/registry/default/avatar-group/avatar-group.tsx](packages/registry/registry/default/avatar-group/avatar-group.tsx) — each stacked avatar (and the `+N` overflow tile) now carries `border-2 border-border/60 ring-2 ring-background` instead of `ring-2 ring-background` alone. (1px was tried first and read as too thin against the 2px page-bg ring; 2px gives the rim a properly visible edge while only insetting the avatar content area by 4px total — barely perceptible on `md` and up.) The ring punches each circle out of its neighbour (unchanged); the hairline keeps the edge readable when the avatar content is close in tone to the page background — e.g. dark transparent-PNG avatars on a dark surface, where the page-color ring and the avatar interior would otherwise blur together. `border-border/60` measures well above the SC 1.4.11 3:1 non-text-contrast minimum in both themes against `--background` and `--card`. The group now also forwards the resolved `size` prop down to each child Avatar (was: only className) so the new `AvatarSizeContext` in `AvatarImage` can correctly gate the blurred-backdrop layer for grouped avatars. The local `avatarSizeClass` import was dropped from `avatar-group.tsx` because the size prop pass-through means Avatar produces its own size className internally — `avatarSizeClass` is still exported from the avatar module for external consumers. Non-breaking.

## 2026-05-15

### Added

- **`InboxMenu` component.** [packages/registry/registry/default/inbox-menu/inbox-menu.tsx](packages/registry/registry/default/inbox-menu/inbox-menu.tsx) — header-level notification center. The trigger is an icon-only `Button` (Bell icon) with a `destructive`-variant unread-count `Badge` overlaid on the top-right corner (capped at `99+` via `maxBadgeCount`); the badge is hidden when unread is zero or `disabled`. The popover content has three slots: a header (title + optional "Mark all read" link that only renders when `onMarkAllRead` is provided AND there is at least one unread item), a scrollable list (`max-h-96`, `divide-y`) of notification rows, and an optional "View all" footer (either an `<a href={viewAllHref}>` or a `<button onClick={onViewAllClick}>`). Each row renders a leading visual (avatar from `avatarUrl` / `avatarFallback`, OR an `Icon` in a tinted-circle, OR a small primary dot for unread items with no visual provided), a `title` (bold when `read` is `false`), an optional 2-line clamped `description`, and a pre-formatted `timestamp` on the right. Rows render as `<a href>` when `item.href` is set or `<button>` otherwise; both surfaces fire `onItemClick(id)`. The trigger's `aria-label` is dynamic (`"Notifications (3 unread)"`) so screen readers get the unread count; the visual badge is `aria-hidden` to avoid double-announcing. When `items` is empty, the popover shows an `EmptyState` (`size="sm"`) with a Bell icon, "No notifications", and "You're all caught up." — both messages overridable via `emptyTitle` / `emptyDescription`. Empty arrays still render the popover trigger; `disabled` blocks the popover from opening. Reference doc: [docs/inbox-menu.md](public/docs/inbox-menu.md). Non-breaking.

- **`TagInput` component.** [packages/registry/registry/default/tag-input/tag-input.tsx](packages/registry/registry/default/tag-input/tag-input.tsx) — controlled `string[]` multi-tag input. The user types into an inline input, presses `Enter` or `,` (configurable via `delimiters`) to commit a chip, and `Backspace` on an empty input removes the last chip. Each committed value renders as a `Badge` with a built-in × remove button (aria-labelled `"Remove <tag>"`). Three modes coexist in one component: (1) free-form (default), (2) free-form + suggestions (`suggestions` prop opens an autocomplete popover anchored to the wrapper as the user types, filtered against already-committed values), (3) allowlist-only (`suggestions` + `allowFreeForm={false}` blocks free-form commits with an inline error). Optional `max` caps the list; once reached the input is disabled and a `Maximum of N tags reached.` hint replaces the placeholder. Optional `validate(value)` returns an error message to block a commit; the message renders inline below the input. Commas inside pasted/typed text split into multiple tags. Duplicates and empty strings are blocked silently. Props: `value`, `onChange`, `suggestions`, `allowFreeForm`, `max`, `validate`, `delimiters`, `placeholder`, `disabled`, `className`, `name`, `id`, `aria-invalid`, `aria-describedby`. The root element receives `data-slot="tag-input"`. Reference doc: [docs/tag-input.md](public/docs/tag-input.md). Non-breaking.

- **`SettingsRow` component.** [packages/registry/registry/default/settings-row/settings-row.tsx](packages/registry/registry/default/settings-row/settings-row.tsx) — pure-layout row for `/settings/*` pages. Composes `title` + optional `description` + control (children) + optional `helperText` + optional `trailing` action slot. Two layouts: `inline` (default, control sits to the right of the label block — good for `Switch`, icon-trigger controls) and `stacked` (control sits on its own row below the label block — good for full-width `Input` / `Textarea` / `SelectPicker`). Save UI is opt-in: by default the row owns no save state at all, so the parent page can run a single batched save flow over multiple rows; when `showSave` is `true`, the row renders a right-aligned `Cancel` + `Save` pair (built on `Button action="cancel"` / `action="save"` so icons and variants stay consistent) and the parent passes `onSave` / `onCancel` / `dirty` / `saving` / `saveLabel` / `cancelLabel` to drive it. The `Cancel` button is only rendered when `onCancel` is provided. The root element receives `data-slot="settings-row"` + `data-layout="<inline|stacked>"` for theming hooks. Reference doc: [docs/settings-row.md](public/docs/settings-row.md). Non-breaking.

- **`ThemeToggle` component.** [packages/registry/registry/default/theme-toggle/theme-toggle.tsx](packages/registry/registry/default/theme-toggle/theme-toggle.tsx) — icon-only `DropdownMenu` trigger that switches the app between three theme modes (`light` / `dark` / `system`). The active mode is reflected by the trigger icon (`Sun` / `Moon` / `Monitor`); the menu uses `DropdownMenuRadioGroup` + `DropdownMenuRadioItem` so only one mode is checked at a time. The choice is persisted to `localStorage` under a configurable key (`storageKey`, default `"lucid-theme"`) and re-applied on mount via `document.documentElement.classList.toggle('dark', …)`. When the mode is `system`, the component subscribes to `window.matchMedia('(prefers-color-scheme: dark)')` so the page also reacts live to OS preference changes. Props: `storageKey`, `align` (forwarded to `DropdownMenuContent`), and all remaining `ButtonProps` except `onClick` / `children` / `asChild` / `action` / `icon` / `loading` (those are owned by the component). Defaults: `variant="outline"`, `size="icon"`, `align="end"`. Also exports a `applyStoredTheme(storageKey?)` named helper that consumers can call once in `main.tsx` before React mounts to avoid a flash of incorrect theme. Replaces the hand-rolled 2-state `apps/demo/src/components/theme-toggle.tsx` that was being reused across the demo's three apps (showcase / saas / pure) — that file is deleted and all four call sites (`App.tsx`, `showcase-sidebar.tsx`, `saas-app.tsx`, `pure-app.tsx`) now import the registry component. Closes a `pure/` registry gap. Reference doc: [docs/theme-toggle.md](public/docs/theme-toggle.md). Non-breaking.

- **Demo: "What's new" page + automatic "New" sidebar badge.** [apps/demo/src/showcase/whats-new-view.tsx](apps/demo/src/showcase/whats-new-view.tsx), [apps/demo/src/showcase/showcase-app.tsx](apps/demo/src/showcase/showcase-app.tsx), [apps/demo/src/showcase/showcase-sidebar.tsx](apps/demo/src/showcase/showcase-sidebar.tsx) — adds a new top-level "What's new" entry to the showcase sidebar (Megaphone icon, anchored above "Use with AI") that opens a page listing recent registry additions and changes grouped by date. Each entry is a clickable card that routes to the corresponding component demo via hash navigation (`href="#/<componentName>"`) so reviewers can jump straight into the API. Entries are tagged with a tone-coded `Badge` (`added` → `success`, `changed` / `fixed` → `info`, `removed` → `destructive`, `breaking` → `destructive`) and pull from a hand-curated `ENTRIES` array at the top of `whats-new-view.tsx`. The sidebar also automatically renders a small "New" badge next to any component whose `ComponentMeta.since` ISO date (YYYY-MM-DD) is within the last 14 days (controlled by `NEW_BADGE_DAYS` in `showcase-sidebar.tsx`). `since` is undefined for older components and set to `'2026-05-14'` on `ThemeToggle`, `SettingsRow`, `TagInput`, and `InboxMenu`. The badge is purely date-driven — no manual flag to toggle, no stale state to clean up. Demo-only change; the published `@pandaworks-sw/lucid-ui` tarball is unaffected. Non-breaking.
